1 Sintesi del Progetto

Il progetto prevede la realizzazione di un portale web per pazienti e medici a supporto di uno studio medico associato composto da 4 medici di medicina generale con circa 1.400 assistiti totali.

Il portale nasce dall'esigenza di ridurre il carico di comunicazioni non strutturate (email, WhatsApp, telefonate) che gravano quotidianamente sullo studio, incanalando le richieste dei pazienti in flussi guidati e tracciati.

Obiettivi principali

  • Permettere ai pazienti di prenotare visite, richiedere ricette e certificati online
  • Fornire ai medici e alla segreteria un pannello di gestione centralizzato per tutte le richieste
  • Eliminare le richieste incomplete grazie a form guidati con campi obbligatori
  • Garantire conformità al GDPR art. 9 (dati sanitari di categoria particolare)
  • Inviare notifiche automatiche (conferme, reminder 24h, aggiornamenti di stato)

2 Architettura del Sistema

Il sistema è basato interamente su servizi Azure PaaS/SaaS ospitati nella region Italy North (Milano), senza macchine virtuali. Non è richiesta gestione di sistema operativo, patch o manutenzione infrastrutturale.

Scelta PostgreSQL vs Azure SQL

È stato scelto PostgreSQL Flexible Server per i seguenti motivi:

  • Costo inferiore rispetto ad Azure SQL a parità di specifiche
  • Nessun vendor lock-in — open source, riproducibile in locale per sviluppo e test
  • Supporto nativo JSONB utile per i campi dinamici dei form (ricette, certificati)
  • Ecosistema Node.js/Prisma è PostgreSQL-first
  • Identico comportamento tra ambiente di sviluppo locale e produzione

5 Flussi Applicativi Principali

5.1 Flusso di Autenticazione e Onboarding Paziente

StepAzioneDettaglio tecnico
1Il paziente sceglie il metodo di accessoTre opzioni: Google OAuth, Microsoft (Entra), Email + OTP
2Autenticazione esternaGoogle/Microsoft: redirect OAuth standard. OTP: generazione codice 6 cifre (TTL 10 min) salvato su PostgreSQL, invio via ACS.
3Primo accesso rilevatoNextAuth callback: se paziente.status = null, creazione record con status = 'pending_approval'.
4Pagina di attesaIl paziente viene reindirizzato a /paziente/pending. Impossibile accedere ad altre route.
5Consenso GDPR (solo primo accesso)Pagina /paziente/gdpr-consent: 2 checkbox obbligatorie. Accettazione salvata su DB con timestamp e IP.
6Notifica alla segreteriaEmail automatica via ACS alla segreteria con link diretto alla pagina approvazioni.
7Approvazione segreteriaPannello admin: la segreteria seleziona il medico di riferimento e approva. Record aggiornato a status = 'active'.
8Notifica al pazienteEmail con link di accesso diretto al portale. Il paziente può ora accedere normalmente.
9Accessi successiviSessione JWT (NextAuth). Middleware verifica status = 'active' ad ogni richiesta. OTP non richiesto.

5.2 Flusso Prenotazione Visita

StepAzioneDettaglio
1Paziente seleziona medico e dataCalendario con slot disponibili da DB. Solo slot aperti (non prenotati) vengono mostrati.
2Invio richiestaRecord su tabella prenotazioni con status = 'pending'. Email di ricezione al paziente.
3Notifica al medico/segreteriaEmail + notifica in-app nel pannello admin (badge rosso contatore).
4Gestione adminSegreteria conferma o rifiuta dalla pagina prenotazioni. Possibilità di inserire note.
5Email al pazienteConferma o rifiuto con motivazione. In caso di conferma: data, ora, indirizzo.
6Reminder automaticoAzure Function (timer trigger ore 08:00): query prenotazioni del giorno seguente → email reminder ai pazienti.
7Cancellazione pazienteConsentita fino a 24h prima. Notifica automatica al medico.

5.3 Flusso Richiesta Ricetta / Certificato

I flussi ricetta e certificato seguono lo stesso pattern: form guidato con campi obbligatori → record su DB → notifica admin → elaborazione manuale → notifica paziente (ricetta pronta / certificato PDF allegato via email).

I certificati vengono generati come PDF lato server (Azure Function + PDFKit) e allegati all'email tramite ACS Attachment API.

6.3 DPIA — Data Protection Impact Assessment

Il trattamento sistematico di dati sanitari per oltre 1.000 interessati richiede la redazione di una DPIA prima del go-live (art. 35 GDPR). La DPIA è prevista come deliverable separato del progetto e include:

  • Descrizione del trattamento e delle finalità
  • Valutazione della necessità e proporzionalità
  • Identificazione e valutazione dei rischi per gli interessati
  • Misure adottate per affrontare i rischi
Nomina DPO Si raccomanda allo studio di verificare con un consulente legale la necessità di nomina di un Data Protection Officer (art. 37 GDPR), probabile dato il trattamento sistematico di dati sanitari su larga scala.

9 Note di Deployment e Manutenzione

9.1 Pipeline CI/CD

Il progetto utilizzerà GitHub Actions per la pipeline CI/CD:

  • Branch main: deploy automatico su ambiente di produzione (App Service)
  • Branch dev: deploy su slot staging di App Service (zero downtime swap)
  • Build: npm run build → artifact Next.js
  • Migrations DB: prisma migrate deploy eseguito pre-deploy

9.2 Backup e Ripristino

  • PostgreSQL Flexible: backup automatico giornaliero con retention 7 giorni (incluso nel servizio)
  • Blob Storage: versioning attivato per i documenti pazienti (referti)
  • RTO stimato: <2h — RPO: 24h (ultimo backup)

9.3 Monitoraggio

  • Application Insights: alert automatici su errori 5xx, latenza >3s, disponibilità <99%
  • Azure Monitor: alert budget a €150/mese
  • Audit trail: consultabile dalla segreteria tramite dashboard Application Insights dedicata
